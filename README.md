# Provenance Guard

**AI201 Project 4 — Content Attribution System**

A backend REST API for creative-sharing platforms. It classifies whether
submitted text was written by a human or generated by AI, scores confidence
in that classification, surfaces a plain-language transparency label to
readers, and handles creator appeals of disputed decisions.

---

## Setup

1. Clone this repo
2. Create and activate a virtual environment:
   ```bash
   python -m venv .venv
   # Windows CMD:
   .venv\Scripts\activate
   # Mac/Linux:
   source .venv/bin/activate
   ```
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Copy `.env.example` to `.env` and add your Groq API key:
   ```
   GROQ_API_KEY=your_key_here
   ```
5. Run the server:
   ```bash
   python app.py
   ```

The API runs at `http://localhost:5000`.

## Architecture Overview

A submitted piece of text takes this path through the system:

```
POST /submit
  │
  ├─ Rate limiter (10/min, 100/hr per IP) → 429 if exceeded
  │
  ├─ Input validation (min 20 chars, max 10,000 chars)
  │
  ├─ classifier.py: run_detection_pipeline()
  │     ├── Signal 1: signal_llm()
  │     │     Groq LLM-as-judge evaluates voice, structure,
  │     │     semantic coherence, linguistic fingerprint
  │     │     → returns score [0–1], label, explanation
  │     │
  │     └── Signal 2: signal_stylometric()
  │           Pure Python: sentence-length SD, TTR, avg word
  │           length, paragraph variance, hedge phrase density
  │           → returns score [0–1], label, explanation
  │
  ├─ final_score = llm × 0.60 + stylometric × 0.40
  │
  ├─ confidence.py: evaluate(final_score)
  │     → verdict: "ai" | "human" | "uncertain"
  │     → confidence score [0–1]
  │     → transparency label dict (3 variants)
  │
  ├─ auditor.py: log_decision()
  │     → writes to SQLite: content_id, timestamp, verdict,
  │       confidence, raw_score, llm_score, stylometric_score,
  │       creator_id, content_preview, status
  │
  └─ JSON response → client
       content_id, attribution, verdict, confidence,
       raw_score, signals, weights, label, status
```

For appeals, the path is:

```
POST /appeal { content_id, creator_reasoning }
  │
  ├─ Rate limiter (30/day per IP)
  ├─ Validate content_id exists and is not already under_review
  ├─ INSERT into appeals table (reason + original_verdict)
  ├─ UPDATE decisions SET status = "under_review"
  └─ JSON response → client { content_id, status, message }
```

The key design principle: every decision is logged before the response is
returned. If the server crashes mid-request, the log is still written.

## Detection Signals

### Why these two signals?

The goal was two signals that are **genuinely independent** — one semantic,
one structural — so that when they disagree, we learn something rather than
just getting noise. An LLM-only system is a black box with no structural
check. A stylometrics-only system is brittle on short texts and misses
semantic patterns entirely. Together they cover different failure modes.

### Signal 1 — LLM-as-judge (Groq, llama-3.3-70b-versatile)

**What it measures:** Semantic and stylistic coherence holistically. The model
evaluates four dimensions: (1) voice — distinctive vs. generic, (2) structure
— organic vs. formulaic, (3) semantic coherence — does it stay safely on-topic
or make unexpected meaningful leaps? (4) linguistic fingerprint — unusual word
choices, syntax quirks, deliberate roughness.

**Why it differs:** AI-generated text is uniformly polished, stays on-topic,
overuses transition phrases ("furthermore," "it is worth noting"), and lacks
genuine personal voice. Human text has tonal shifts, unexpected metaphors,
rough patches, and personal references the model would have no reason to invent.

**Blind spot:** The judge is itself an AI — it may have systematic biases
about what "sounds human." A human who writes in a clean academic style may
be rated as AI-like. Also unreliable on texts under ~80 words.

**Weight: 60%** — primary signal; captures semantic properties that
statistics alone cannot see.

---

### Signal 2 — Stylometric heuristics (pure Python, no external libs)

**What it measures:** Five statistical surface features:

| Feature | AI pattern | Human pattern |
|---|---|---|
| Sentence-length SD | Low (< 5 words) — uniform | High (> 8) — variable |
| Type-Token Ratio (TTR) | Extreme (< 0.35 or > 0.85) in long texts | Moderate 0.50–0.75 |
| Average word length | 4.8–5.5+ chars (formal/latinate) | 3.8–4.7 chars |
| Paragraph length variance | Low SD — uniform structure | High SD — jagged |
| Hedge phrase density | > 0.5 per 100 words | Near 0 |

**Why it differs:** LLMs are trained on averaged human output and optimize
for coherence — their statistical fingerprint is more regular and "average."
Real human writers are idiosyncratic: they vary sentence length dramatically,
repeat words for rhythmic effect, and write uneven paragraphs.

**Blind spot:** A haiku poet or flash-fiction writer with naturally uniform
short lines will look AI-like. Academic writing also has low variance. Short
texts (< 150 words) don't yield reliable TTR readings — the TTR feature is
skipped for short texts to prevent noise.

**Weight: 40%** — independent structural check that is not affected by
semantic content.

---

### How they combine

```
final_score = llm_score × 0.60 + stylometric_score × 0.40
```

LLM gets higher weight because semantic judgment is harder to fake. The
stylometric signal acts as a moderating force: when the LLM is confidently
wrong (e.g., it reads a highly polished human essay as AI), the stylometrics
often show human-like variance and pull the combined score toward uncertain —
which is the right outcome.

## Confidence Scoring

### How it works

Raw pipeline score is [0, 1] where 1.0 = maximally AI-like.

**Verdict thresholds** (asymmetric by design):

| raw_score | verdict | label shown |
|---|---|---|
| ≥ 0.75 | `ai` | ⚠️ Likely AI-Generated |
| ≤ 0.25 | `human` | ✅ Likely Human-Written |
| 0.25–0.75 | `uncertain` | 🔍 Authorship Unclear |

**Confidence formula:**
```
confidence = abs(raw_score - 0.5) × 2
```
- `raw_score = 0.50` → confidence = 0.00 (maximum uncertainty)
- `raw_score = 1.00` → confidence = 1.00 (maximum certainty)

### Why the wide uncertain band?

A false positive (labeling a human's work as AI) causes real reputational
harm to a creator. A false negative (missing AI content) misleads readers
but harms no individual directly. The 50%-wide uncertain band biases the
system toward caution. You need substantial evidence — not just a slight
lean — before the system makes a confident accusation.

### Two example submissions showing meaningful variation

**Example A — High confidence AI** (hedge-phrase paragraph):

> *"Artificial intelligence represents a transformative paradigm shift in
> modern society. It is important to note that while the benefits of AI are
> numerous, it is equally essential to consider the ethical implications.
> Furthermore, stakeholders across various sectors must collaborate."*

```
LLM score:         0.90  (ai)
Stylometric score: 0.58  (uncertain)
final_score:       0.772
verdict:           ai
confidence:        0.544
label:             ⚠️ Likely AI-Generated — Confidence: 54%
```

**Example B — Low confidence (uncertain)** (borderline formal academic):

> *"The relationship between monetary policy and asset price inflation has
> been extensively studied. Central banks face a fundamental tension between
> price stability and the unintended consequences of prolonged low rates."*

```
LLM score:         0.80  (ai)
Stylometric score: 0.47  (uncertain)
final_score:       0.668
verdict:           uncertain
confidence:        0.336
label:             🔍 Authorship Unclear — Detection score: 67%
```

The difference is meaningful: Example A is confidently labeled AI (0.772),
Example B is not labeled at all (0.668 stays under the 0.75 threshold) even
though the LLM also rated it highly. The stylometric signal's disagreement
pushed it to uncertain — protecting the human academic writer from a false
accusation.

### How I validated the scores are meaningful

Ran 4 required spec inputs and confirmed separation:

| Input | raw_score | verdict |
|---|---|---|
| Clearly AI (hedge phrases) | 0.772 | ai |
| Clearly human (casual ramen review) | 0.156 | human |
| Borderline formal academic | 0.668 | uncertain |
| Borderline lightly-edited AI | 0.540 | uncertain |

Gap between clearly-AI and clearly-human: **0.616**. Both borderlines land
in uncertain rather than being forced to a wrong verdict. A score of 0.51
and 0.95 produce genuinely different outputs — not the same label.

## Rate Limiting

Implemented via `flask-limiter`, keyed by remote IP address.

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | 10/min, 100/hr | A genuine creator submits work occasionally — not dozens of times per minute. 10/min is generous for legitimate use but stops automated flood attacks cold. 100/hr gives prolific creators a meaningful daily budget. |
| `POST /appeal` | 30/day | Appeals are deliberate manual actions. A real creator would file at most 2–3 per day. 30/day is generous for humans, restrictive for scripts. |

**Rate limit verification** — 12 rapid requests (limit: 10/min):

## Appeals Workflow

1. Creator submits `POST /appeal` with `content_id` + `creator_reasoning`
2. System validates: content_id exists, not already under review
3. Validates: reason is at least 10 characters
4. Records appeal in `appeals` table (reason + original verdict + timestamp)
5. Updates decision status to `under_review`
6. Returns `{ content_id, status: "under_review", message }`

No automated re-classification occurs. Re-running the same signals that
produced the original (possibly flawed) verdict would not fix a false
positive. A human moderator reviews the appeal and the creator's reasoning.

Duplicate appeals are rejected with 400 "already under review."

## Known Limitations

**1. Minimalist poetry and uniform prose are misread by stylometrics.**

A poet who naturally writes short, consistent-length lines — haiku, imagist
verse, flash fiction — will produce low sentence-length SD and low paragraph
variance. The stylometric signal scores this as AI-like. If the LLM agrees
(e.g., it also reads the spare language as "overly uniform"), the piece gets
labeled AI-Generated even though it's deeply personal human work. This is
a direct consequence of the stylometric signal using uniformity as an AI
indicator — a reasonable heuristic for typical prose, but wrong for
intentionally sparse styles.

The wide uncertain band mitigates this: in practice the LLM usually detects
poetic voice and scores it human-like, pulling the combined score into
uncertain rather than ai. But it's a real gap.

**2. Deliberately humanized AI output defeats both signals.**

A user who prompts an AI with "write this in a raw, personal, slightly messy
style with sentence fragments and emotional language" can produce text that
the LLM rates as human-like and that has stylistic irregularity. Neither
signal was designed to detect "AI mimicking human imperfection." The system
would likely return uncertain or even human for such content. This is a
fundamental limitation of behavioral signals — they measure output properties,
not provenance.

**3. Academic and technical human writing is stylometrically AI-like.**

Long latinate words, hedge phrases ("it has been argued"), uniform paragraph
structure — these are features of formal academic writing regardless of
authorship. A PhD student's abstract may score 0.65–0.70 on stylometrics,
pushing the combined score toward uncertain even when the LLM correctly
identifies it as human. The system handles this by landing in uncertain
rather than making a wrong call, but it means academic content can rarely
get a confident "human" verdict.

## Spec Reflection

**One way the spec helped:**
Writing out the three transparency label variants verbatim in `planning.md`
before building anything forced a concrete design decision early: the appeal
note should appear on all three variants, not just the AI label. Without
writing out the full text, that would have been easy to miss — the appeal
note on the "human" label matters for cases where the system is correct but
the creator disputes it anyway.

**One way implementation diverged from the spec:**
The spec defined `MIN_CONTENT_CHARS = 50`. During testing with the CodePath
example curl command (`"The sun dipped below the horizon."` — 34 chars),
this rejected valid demo inputs. The minimum was lowered to 20 characters.
This is the right call for a creative platform where a short poem or a
two-line excerpt is a legitimate submission, and the 50-char floor was
arbitrary rather than principled.

## AI Usage

**Instance 1 — LLM system prompt design**

I directed the AI tool to generate the system prompt for `signal_llm()` —
specifically asking it to produce a prompt that evaluates four named dimensions
(voice, structure, semantic coherence, linguistic fingerprint) and returns
output in a structured `SCORE / LABEL / EXPLANATION` format.

What it produced was reasonable but had one failure mode: when the score and
label disagreed (e.g., `SCORE: 0.80` with `LABEL: human`), the parser had
no way to know which to trust. I added a consistency check that snaps the
label to match the score when they diverge badly — the spec didn't include
this, I added it after seeing the first few test outputs produce contradictory
fields.

**Instance 2 — Stylometric TTR threshold**

I asked the AI tool to help design the TTR (Type-Token Ratio) scoring
thresholds — what TTR values indicate AI vs. human text in creative prose.

It suggested flagging both very high and very low TTR as AI signals, which
is correct for long texts. What it missed: for texts under 150 words, TTR
is naturally high (a 50-word paragraph has ~85% unique words regardless of
authorship) because there's simply not enough room for word repetition. I
discovered this during Milestone 4 calibration when all four test texts
(all under 150 words) had TTR ≥ 0.85 and were being incorrectly penalized.
The fix — skip TTR for short texts — was mine, not the AI's suggestion.

## Stretch Feature: Ensemble Detection

Pass `"ensemble": true` in the submit request body to activate 3-signal mode,
which adds a perplexity proxy signal:

| Signal | Weight |
|---|---|
| LLM-as-judge | 50% |
| Stylometric | 30% |
| Perplexity proxy | 20% |

The perplexity proxy estimates word-choice predictability using a bigram
branching factor and safe-word ratio. AI text tends to stay "on the
probability curve." This is a lightweight approximation, not true perplexity,
but it captures a structural dimension independent of the other two signals.
