# Provenance Guard

**AI201 Project 4 — Content Attribution System**

A backend API for creative-sharing platforms that classifies whether submitted
text content was written by a human or generated by AI, scores confidence in
that classification, surfaces a transparency label to readers, and handles
creator appeals.

---

## Setup

1. Clone this repo
2. Create and activate a virtual environment:
   ```bash
   python -m venv .venv
   # Windows CMD:
   .venv\Scripts\activate
   # Mac/Linux:
   source .venv/bin/activate
   ```
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Copy `.env.example` to `.env` and add your Groq API key:
   ```
   GROQ_API_KEY=your_key_here
   ```
5. Run the server:
   ```bash
   python app.py
   ```

The API runs at `http://localhost:5000`.

## Detection Signals

### Signal 1 — LLM-as-judge (Groq, llama-3.3-70b-versatile)

**What it captures:** Semantic and stylistic coherence holistically — voice,
structure, linguistic fingerprint, and whether the text feels "on a probability
curve" vs. genuinely idiosyncratic. The LLM evaluates four dimensions: voice,
structure, semantic coherence, and linguistic fingerprint.

**Why it differs:** AI-generated text is uniformly polished, stays on topic,
overuses hedge phrases, and lacks genuine personal voice or unexpected leaps.
Human text has rough patches, tonal shifts, and personal references.

**Blind spot:** Can be fooled by humans who write in a clinical style, or by
AI text that deliberately mimics personal voice. Very short texts give it
little signal.

**Weight:** 60%

---

### Signal 2 — Stylometric heuristics (pure Python)

**What it captures:** Five statistical surface features:
- **Sentence-length SD** — AI text has lower variance (more uniform sentences)
- **Type-Token Ratio (TTR)** — unique_words / total_words; extremes signal AI
- **Average word length** — AI prose skews slightly longer/more formal
- **Paragraph length variance** — AI tends toward uniform paragraph structure
- **Hedge phrase density** — counts phrases like "it's worth noting" per 100 words

**Why it differs:** Language models are trained on averaged human output,
making their statistical fingerprint more "average" — less variance, more
regularity. Human writers are idiosyncratic.

**Blind spot:** Poets who naturally write short consistent lines, technical
writers, or anyone whose style is naturally uniform will look AI-like.
Short texts don't yield meaningful statistics.

**Weight:** 40%

---

### How the two signals combine

```
final_score = llm_score × 0.60 + stylometric_score × 0.40
```

LLM gets higher weight because it captures semantic properties that
stylometrics cannot see. Stylometrics acts as an independent check —
it may catch cases where the LLM was deceived by well-crafted language.

## Confidence Scoring

The raw pipeline score is a float [0, 1] (1.0 = maximally AI-like).

**Verdict thresholds:**

| raw_score range | verdict     |
|-----------------|-------------|
| ≥ 0.75          | `ai`        |
| ≤ 0.25          | `human`     |
| 0.25–0.75       | `uncertain` |

The uncertain band is intentionally wide (50% of the full range). A score of
0.51 produces "Authorship Unclear." A score of 0.95 produces "Likely
AI-Generated" with ~97% confidence shown to the reader. These are meaningfully
different outcomes for meaningfully different scores.

**Why the asymmetry?** A false positive (labeling a human's work as AI) is
more harmful than a false negative. Creators suffer real reputational damage
from misclassification; readers experiencing a missed AI label have a much
smaller harm. The wide uncertain band protects creators.

**Confidence formula:**
```
confidence = abs(raw_score - 0.5) × 2
```
- `0.0` = maximum uncertainty (raw_score = 0.5)
- `1.0` = maximum certainty (raw_score = 0.0 or 1.0)

## Rate Limiting

Implemented via `flask-limiter`, keyed by remote IP address.

| Endpoint  | Limit            | Reasoning |
|-----------|------------------|-----------|
| `POST /submit` | 10/min, 100/hr | A genuine creator submits work occasionally, not dozens of times per minute. 10/min is generous for legitimate use but throttles automated flood attacks. 100/hr gives prolific creators a daily buffer without enabling abuse. |
| `POST /appeal` | 30/day         | Appeals are deliberate manual actions. Even a very active creator would rarely file more than 2–3 appeals per day. 30/day is generous for humans, restrictive for scripts. |

Exceeding a limit returns HTTP 429 with a standard Flask-Limiter error response.

## Appeals Workflow

1. Creator submits `POST /appeal` with their `content_id` and explanation
2. System validates the content_id exists in the audit log
3. Checks the decision isn't already under review
4. Records the appeal (reason + original verdict) in the `appeals` table
5. Updates the decision status to `under_review`
6. Returns confirmation to the creator

No automated re-classification occurs — a human moderator reviews the appeal.
This is intentional: automated re-classification using the same flawed signals
that produced the original decision would not actually fix false positives.

## Stretch Feature: Ensemble Detection

Pass `"ensemble": true` in the submit request body to activate 3-signal mode.

A third signal (perplexity proxy) is added:

| Signal            | Weight |
|-------------------|--------|
| LLM-as-judge      | 50%    |
| Stylometric       | 30%    |
| Perplexity proxy  | 20%    |

**Perplexity proxy:** Estimates word-choice predictability using a bigram
branching factor and safe-word ratio. AI text tends to stay "on the
probability curve" — word choices are safer and more expected. This is a
lightweight approximation of perplexity, not a true language-model calculation,
but it adds an independent structural dimension.
