# Planning: Provenance Guard

**AI201 Project 4 — Content Attribution System**

---

## Problem Statement

Creative platforms (writing, music, art sharing) need a way to give readers
context about whether posted content was written by a human or generated by AI.
The goal is not to police creativity — it is to protect attribution, build
reader trust, and give creators a fair path to contest decisions.

Provenance Guard is a backend REST API that any platform can plug into to:
1. Classify submitted text content using multiple independent signals
2. Score confidence in that classification (not just a binary yes/no)
3. Surface a plain-language transparency label to readers
4. Handle creator appeals of disputed classifications
5. Log every decision in a structured audit trail

### Submission Flow (step by step)

```
Client
  │
  │  POST /submit  { "content": "...", "ensemble": false }
  ▼
[Rate limiter: 10/min, 100/hr per IP]  →  HTTP 429 if exceeded
  │
  │  raw text string
  ▼
classifier.py :: run_detection_pipeline(content)
  │
  ├──► Signal 1: signal_llm(content)
  │      ↳ Groq API call (llama-3.3-70b-versatile)
  │      ↳ Returns: { score: 0.0–1.0, label: "ai/human/uncertain", explanation: str }
  │      ↳ score=1.0 means "definitely AI-generated"
  │
  ├──► Signal 2: signal_stylometric(content)
  │      ↳ Pure Python — no API call
  │      ↳ Measures: sentence-len SD, TTR, avg word len, paragraph variance, hedge density
  │      ↳ Returns: { score: 0.0–1.0, label: "ai/human/uncertain", explanation: str }
  │
  │  final_score = (llm_score × 0.60) + (stylometric_score × 0.40)
  │  [range: 0.0–1.0, where 1.0 = maximally AI-like]
  │
  ▼
confidence.py :: evaluate(final_score)
  │
  │  verdict:    final_score ≥ 0.75 → "ai"
  │              final_score ≤ 0.25 → "human"
  │              else              → "uncertain"
  │
  │  confidence: abs(final_score - 0.5) × 2   [0.0=max uncertainty, 1.0=max certainty]
  │
  │  label:      dict with heading, badge, body, detail, appeal_note
  │              (three variants — see Transparency Label Design section)
  │
  ▼
auditor.py :: log_decision(content_id, verdict, confidence, raw_score, ...)
  │
  │  Writes one row to SQLite decisions table
  │  Prints one-line summary to terminal
  │
  ▼
HTTP 200 JSON response:
  {
    content_id, verdict, confidence, raw_score,
    signals: { llm: {...}, stylometric: {...} },
    weights: { llm: 0.6, stylometric: 0.4 },
    label: { heading, badge, icon, body, detail, appeal_note }
  }
```

### Narrative Summary

A submitted piece of text flows through two independent detection signals
(LLM semantic judgment + stylometric statistics), gets combined into a
single weighted score, and is converted to a verdict, confidence value, and
plain-language transparency label — all logged to SQLite before the response
is returned. An appeal routes through a separate module that validates the
content_id, records the creator's reasoning alongside the original decision,
and flips the status to "under_review" for human moderator review.

## Q1 — Detection Signals

### Signal 1 — LLM-as-judge (Groq / llama-3.3-70b-versatile)

**What it measures:**
Semantic and stylistic coherence as a whole. The model evaluates four
dimensions: (1) voice — is it distinctive or generic? (2) structure — organic
or formulaic? (3) semantic coherence — does it stay safely on-topic? (4)
linguistic fingerprint — are there surprising word choices or syntax quirks?

**Output format:**
```
SCORE: 0.87
LABEL: ai
EXPLANATION: Text exhibits uniform paragraph structure and overuse of
hedge phrases typical of AI generation.
```
Parsed into `{ score: float, label: str, explanation: str }`.
Score of 1.0 = definitely AI, 0.0 = definitely human.

**Why it differs between human and AI text:**
AI-generated text is uniformly polished, stays on topic, uses transition
phrases ("furthermore," "it is worth noting"), and lacks genuine personal
voice. Human text has tonal shifts, unexpected metaphors, rough patches,
and personal references the model would have no reason to invent.

**Blind spot:**
The LLM judge is itself an AI — it may have systematic biases about what
"sounds human." A human who writes in a clean, formal style may be flagged.
An AI given a "write in a raw, personal voice" prompt may fool it. Also
ineffective on texts under ~80 words.

**Weight: 60%** — primary signal; captures semantic properties that
statistics alone cannot see.

---

### Signal 2 — Stylometric heuristics (pure Python, no external libs)

**What it measures:**
Five statistical surface features computed directly from the text:

| Feature | AI pattern | Human pattern |
|---|---|---|
| Sentence-length SD | Low (< 5 words) — uniform sentences | High (> 8 words) — variable length |
| Type-Token Ratio (TTR) | Extreme high or low (< 0.35 or > 0.85) | Moderate 0.50–0.75 |
| Average word length | 4.8–5.5+ chars (formal/latinate) | 3.8–4.7 chars |
| Paragraph length variance | Low SD (uniform paragraphs) | High SD (jagged structure) |
| Hedge phrase density | > 0.5 per 100 words | Near 0 |

**Output format:**
`{ score: float, label: str, explanation: str }`
Score of 1.0 = maximally AI-like statistics, 0.0 = maximally human-like.
The explanation includes the raw feature values (e.g. `sent_len_sd=3.2, ttr=0.82`).

**Why it differs:**
LLMs are trained on averaged human output and optimize for coherence —
this makes their statistical fingerprint more "average" and regular.
Real human writers are idiosyncratic: they vary sentence length
dramatically, repeat words for rhythmic effect, and write uneven paragraphs.

**Blind spot:**
A haiku poet or flash-fiction writer who naturally uses short uniform lines
will look AI-like. Technical/legal/academic human writing also has low
variance. Short texts (< 50 words) don't yield enough data points. Hedge
phrase detection is English-only.

**Weight: 40%** — independent of semantic judgment; catches what the LLM misses.

---

### How the two signals combine

```
final_score = (llm_score × 0.60) + (stylometric_score × 0.40)
```

Both signals run independently and in parallel. The LLM gets higher weight
because it captures meaning-level patterns; stylometrics acts as a
structural check that is independent of semantic content. If the two signals
strongly disagree, the result is typically pushed toward "uncertain" — which
is the correct outcome when evidence is mixed.

Example: LLM says 0.30 (human-like), stylometrics says 0.65 (AI-like).
Combined: 0.30×0.60 + 0.65×0.40 = 0.44 → falls in the uncertain band.
The label reads "Authorship Unclear." No false accusation is made.

---

### Signal 3 — Perplexity proxy (stretch / ensemble mode only)

Activated by passing `"ensemble": true`. Uses bigram branching factor and
safe-word ratio to approximate how predictable word choices are. AI text
stays "on the probability curve"; human text deviates more often.

Weights in ensemble mode: LLM 50%, Stylometric 30%, Perplexity proxy 20%.

## Q2 — Uncertainty Representation

### What a score of 0.6 means

`raw_score = 0.60` falls in the uncertain band (0.25–0.75).
Verdict: **"uncertain"**. Label: "Authorship Unclear."

The system is saying: "We have a weak lean toward AI-like patterns, but
not enough evidence to make a confident claim either way. Both signals
gave mixed results." A 0.60 score is meaningfully different from 0.90.

### Score-to-verdict mapping

```
raw_score ≥ 0.75  →  verdict = "ai"       →  "Likely AI-Generated"
raw_score ≤ 0.25  →  verdict = "human"    →  "Likely Human-Written"
0.25 < score < 0.75 → verdict = "uncertain" → "Authorship Unclear"
```

The uncertain band covers 50% of the full range. This is intentional.
A false positive (labeling a human as AI) causes real reputational harm
to a creator. The system biases toward "uncertain" to protect creators.
A false negative (missing AI content) misleads readers but harms no
individual directly.

### Confidence score formula

```
confidence = abs(raw_score - 0.5) × 2
```

| raw_score | verdict   | confidence | Meaning                         |
|-----------|-----------|------------|---------------------------------|
| 0.50      | uncertain | 0.00       | Completely 50/50, no signal     |
| 0.60      | uncertain | 0.20       | Weak lean toward AI             |
| 0.74      | uncertain | 0.48       | Strong lean but not over thresh |
| 0.75      | ai        | 0.50       | Just crossed the AI threshold   |
| 0.92      | ai        | 0.84       | High-confidence AI verdict      |
| 0.10      | human     | 0.80       | High-confidence human verdict   |

A score of 0.51 and a score of 0.95 produce meaningfully different outputs:
- 0.51 → verdict="uncertain", badge="UNVERIFIED", confidence=0.02
- 0.95 → verdict="ai", badge="AI-GENERATED", confidence=0.90, shown as "~95% confidence"

## Q3 — Transparency Label Design

The label is a dict with six fields. All three variants are written out
exactly as they appear in API responses.

### Variant 1 — High-Confidence AI (raw_score ≥ 0.75)

```
heading:      "⚠️ Likely AI-Generated"
badge:        "AI-GENERATED"
icon:         "🤖"
body:         "Our system found strong signals suggesting this content was
               generated by an AI tool rather than written by a human author.
               Confidence: [N]%."
detail:       "This determination is based on a combination of language-pattern
               analysis and stylometric features. It is not infallible."
appeal_note:  "If you are the creator and believe this is incorrect, you can
               submit an appeal to have your content reviewed by a human moderator."
```

### Variant 2 — High-Confidence Human (raw_score ≤ 0.25)

```
heading:      "✅ Likely Human-Written"
badge:        "HUMAN-WRITTEN"
icon:         "✍️"
body:         "Our system found strong signals suggesting this content was
               written by a human author. Confidence: [N]%."
detail:       "This determination is based on a combination of language-pattern
               analysis and stylometric features. No automated system is perfect."
appeal_note:  "Attribution labels can be appealed if you believe the result
               is inaccurate."
```

### Variant 3 — Uncertain (0.25 < raw_score < 0.75)

```
heading:      "🔍 Authorship Unclear"
badge:        "UNVERIFIED"
icon:         "❓"
body:         "Our system could not determine with confidence whether this
               content was written by a human or generated by AI. The signals
               we measure gave mixed or inconclusive results."
detail:       "Detection score: [N]%. A score near 50% means our signals did
               not strongly point in either direction."
appeal_note:  "Creators can submit additional context or request human review
               to clarify the attribution of this content."
```

### Design rationale

The "uncertain" label is intentionally neutral — it does not say "probably
AI" or even hint at a direction. A creator whose haiku was misread by the
stylometric signal sees "Authorship Unclear," not an accusation. The appeal
note is present on ALL three variants so creators always know their options.

## Q4 — Appeals Workflow

### Who can appeal

Any creator who submitted content and received a content_id. There is no
authentication in this project — the content_id is the access token.
(A production system would tie this to user accounts.)

### What they provide

```json
{
  "content_id": "uuid from the original /submit response",
  "reason":     "Free-text explanation, minimum 10 characters.
                 e.g.: I wrote this poem over three weeks. The short lines
                 are intentional haiku structure, not AI uniformity."
}
```

### What the system does on receipt

1. Validate: content_id exists in the decisions table
2. Validate: status is not already "under_review" (no duplicate appeals)
3. Validate: reason is at least 10 characters
4. INSERT one row into the appeals table:
   `(content_id, timestamp, creator_reason, original_verdict, status="under_review")`
5. UPDATE decisions SET status = "under_review" WHERE content_id = ?
6. Return success JSON with new_status = "under_review"

### What a human reviewer would see (GET /log)

```json
{
  "id": 3,
  "content_id": "i9j0k1l2-...",
  "timestamp": "2026-06-29T10:15:32Z",
  "verdict": "ai",
  "confidence": 0.78,
  "raw_score": 0.89,
  "llm_score": 0.91,
  "stylometric_score": 0.86,
  "content_preview": "First 300 chars of the submitted text...",
  "status": "under_review"
}
```

The reviewer sees: original verdict, all signal scores (to understand why
the system decided what it did), the content preview, and the appeal reason.
They can then mark it resolved outside the API (or via a future endpoint).

### What is NOT automated

No automatic re-classification occurs. Using the same signals that produced
the original (possibly flawed) verdict to re-classify would not fix anything.
A human reviewer is required to resolve appeals.

## Q5 — Anticipated Edge Cases

### Edge case 1 — Haiku / minimalist poetry with intentional repetition

A human poet writes in a deliberately sparse, repetitive style:
```
the rain falls again
the rain falls and falls again
again the rain falls
```
Stylometric signal: sentence-length SD is near zero (all ~5 words). TTR is
very low (many repeated words). Hedge density: zero. This profile scores
highly AI-like on stylometrics (~0.70). The LLM signal may correctly identify
the poetic voice and score it human-like (~0.25). Combined: ~0.43 → uncertain.

**Mitigation:** Wide uncertain band means no false AI accusation. Creator
gets "Authorship Unclear" and an appeal path.

---

### Edge case 2 — AI text with deliberate "humanization" prompting

A user generates content with a prompt like "write this in a raw, personal,
slightly messy style with sentence fragments and irregular punctuation." The
result may fool the LLM signal (it reads as human-like voice). The
stylometric signal may still catch the underlying uniformity depending on
the output. This could produce a false negative (we label it uncertain or
even human when it's AI).

**Mitigation:** We acknowledge in the label that no automated system is
perfect. The transparency label is informational, not a legal determination.

---

### Edge case 3 — Very short content (< 100 words)

A tweet-length poem or a four-line excerpt gives both signals very little
to work with. The stylometric signal has too few sentences to compute
meaningful SD. The LLM signal has too little voice or structure to evaluate.
Both will tend toward 0.5 (uncertain).

**Mitigation:** Minimum content length of 50 characters is enforced at the
API level. Short content will almost always land in "uncertain" which is
the honest answer — we simply don't have enough signal.

---

### Edge case 4 — Human writing in a formal/academic register

A PhD student submits their abstract: long latinate words, hedge phrases
("it is worth noting"), uniform paragraph structure, high TTR. The stylometric
signal will score this as highly AI-like. The LLM signal may also rate it
AI-like due to the formal register.

**Mitigation:** Again, the uncertain band protects borderline cases. But this
is a real limitation — stylometrics are not well-calibrated for academic
register. Documented as a known blind spot.

## Rate Limiting

| Endpoint | Limit | Reasoning |
|---|---|---|
| POST /submit | 10/min, 100/hr | A genuine creator submits work occasionally, not dozens of times per minute. 10/min is generous for legitimate use but throttles automated flood attacks. 100/hr gives prolific creators a daily buffer. |
| POST /appeal | 30/day | Appeals are deliberate manual actions. A genuine creator would rarely file more than 2–3 appeals per day. 30/day is generous for humans, restrictive for scripts. |

## Data Model

**decisions table (SQLite):**

| Field | Type | Description |
|---|---|---|
| id | INTEGER PK | Auto-increment |
| content_id | TEXT | UUID assigned at submission |
| timestamp | TEXT | ISO 8601 UTC |
| verdict | TEXT | "ai" / "human" / "uncertain" |
| confidence | REAL | 0.0–1.0 confidence score |
| raw_score | REAL | Combined pipeline score 0.0–1.0 |
| llm_score | REAL | LLM signal score |
| stylometric_score | REAL | Stylometric signal score |
| content_preview | TEXT | First 300 chars of content |
| status | TEXT | "pending" / "under_review" / "resolved" |

**appeals table (SQLite):**

| Field | Type | Description |
|---|---|---|
| id | INTEGER PK | Auto-increment |
| content_id | TEXT | References decisions.content_id |
| timestamp | TEXT | ISO 8601 UTC |
| creator_reason | TEXT | Free-text explanation from creator |
| original_verdict | TEXT | Verdict at time of appeal |
| status | TEXT | "under_review" / "resolved" |
